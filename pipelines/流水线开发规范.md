# 流水线开发规范

## 1. 目的与适用范围

本文档用于统一 `ar/allrun` 流水线在**设计、开发、打包、加载、执行、运维**各阶段的实现标准，适用于：

- `pipelines/` 下所有流水线工程；
- `backend/pkg/pipeline` 执行模型对应的模板与镜像；
- GraphQL 与 CLI 两条调用链（`run/stop/resume/load/build`）。

本文档以代码当前行为为准；如与旧设计文档不一致，按后端实现约束落地。

## 3. 关键术语

- **流水线模板**：`<pipelineName>.template.json`，定义步骤与 DAG 依赖。
- **步骤（step）**：模板数组中的一个对象，最终映射为一次 OCI 容器执行。
- **DAG 边**：`step.nodes` 表示当前步骤指向的后继步骤（`name -> nodes[]`）。
- **任务（task）**：一次 `run` 产生的运行实例，唯一标识为 `taskId`。
- **运行目录（runDir）**：`/var/lib/ar/tasks/<pipelineName>/<taskId>/`。
- **执行快照**：`runDir/pipeline.json`，记录渲染后的步骤及状态。

## 4. 目录与交付物规范

### 4.1 流水线工程目录（推荐）

建议每条流水线在 `pipelines/<pipeline-id>/` 下包含：

```text
pipelines/<pipeline-id>/
├── <pipeline-id>.template.json
├── images.txt
├── Makefile
├── design.md
├── metadata/
│   ├── versions.lock
│   └── manifest.json
├── scripts/
│   ├── download/
│   ├── package/
│   └── remote/
├── templates/
│   ├── config/
│   ├── manifests/
│   ├── pki/
│   └── systemd/
└── steps/
    └── <step-image>/
```

### 4.2 必备交付物

- 必须有且仅有一个主模板：`<pipelineName>.template.json`。
- 必须维护 `images.txt`：声明构建流水线镜像时要拉取/打包的镜像列表。

---

## 5. 命名规范

### 5.1 `pipelineName` 命名

- 必须使用 ASCII 安全字符：`[a-zA-Z0-9._-]`。
- 禁止使用空格、路径分隔符、中文或特殊字符作为正式 `pipelineName`。
- 模板文件名必须为：`<pipelineName>.template.json`。

> 原因：后端会对流水线名做 `sanitize`，非法字符会被剔除，可能导致找不到模板或运行目录异常。

### 5.2 `step.name` 命名

- 必须全局唯一（同一模板内不可重名）。
- 推荐使用小写中划线命名（如 `global-install-containerd-all`）。
- 虽然后端可回退容器 ID（`stepN`），但不应依赖回退机制。

### 5.3 镜像命名

- 模板中的 `image` 建议使用完整引用（`repo/name:tag`）。
- 版本必须固定，禁止无控制漂移（生产禁用未审计 `latest`）。

---

## 6. 模板结构规范（`*.template.json`）

### 6.1 顶层结构

- 顶层必须是**非空 JSON 数组**。
- 每个元素结构：
  - `name`：必填；
  - `image`：必填；
  - `entrypoint`：可选；
  - `args`：可选；
  - `env`：可选；
  - `nodes`：可选（后继步骤名列表）。

### 6.2 DAG 规则

- `nodes` 表示**后继节点列表**，不是前置依赖列表。
- `nodes` 中每个名称必须能在模板中找到对应 `step.name`。
- 必须保证 DAG 无环；有环会导致执行阶段拓扑排序失败。

### 6.3 `entrypoint/args/env` 规则

- `entrypoint` 为空时，回退使用镜像默认 Entrypoint。
- 若 `entrypoint + args` 仍为空，则回退镜像 Cmd；仍为空即执行失败。
- `env` 格式必须是 `KEY=VALUE` 字符串数组。

### 6.4 模板渲染上下文（Go `text/template`）

运行时可使用 `.nodes`，每项可访问：

- `.IP`
- `.Port`
- `.Username`
- `.Password`
- `.LabelsStr`（`k=v,k2=v2`）

支持模板函数：

- `len`（支持 `.nodes` / `[]string`）
- `join`（用法：`{{join "," .someStringArray}}`）
- `labelHas`（用法：`{{labelHas .LabelsStr "etcd" "true"}}`）
- `ipsByLabel`（用法：`{{join "," (ipsByLabel .nodes "etcd" "true")}}`）

> 推荐将与节点相关的展开逻辑统一放在 `env` 字段，降低 `args` 拼接复杂度。

---

## 7. 节点输入规范

### 7.1 支持格式

`-n` 节点文件支持两种 JSON：

1. 包装对象：`{"nodes":[...]}`
2. 直接数组：`[{...}]`

### 7.2 节点字段

- `ip`：必填
- `port`：可选
- `username`：必填
- `password`：必填
- `labels`：`[{key,value}]`

建议标签语义遵循：

- `role=master|worker`
- `registry=true|false`

### 7.3 调用接口规范（CLI / GraphQL）

- CLI 入口：
  - `allrun pipeline load`
  - `allrun pipeline run`
  - `allrun pipeline task stop`
  - `allrun pipeline task resume`
  - `allrun pipeline task log`
- GraphQL 入口：
  - `runPipeline(input: RunPipelineInput!)`
  - `stopPipeline(taskId: String!)`
  - `resumePipeline(taskId: String!)`
- 两条调用链必须共用同一模板与任务状态文件（`pipeline.json`）语义，不允许定义分叉状态模型。

---

## 8. 运行时目录与挂载规范

### 8.1 宿主机目录（默认）

- `--pipelines-dir`：`/var/lib/ar/pipelines`
- `--images-store-dir`：`/var/lib/ar/images`
- `--oci-runtime-root`：`/var/lib/ar/runc`
- 任务目录：`/var/lib/ar/tasks/<pipelineName>/<taskId>/`

### 8.2 单次任务目录结构

```text
runDir/
├── pipeline.json
├── node1/
├── node2/
├── ...
├── bundles/<stepName>/
└── logs/<containerID>.stdout|stderr
```

### 8.3 步骤容器挂载点

- `/tasks` -> `runDir`
- `/current-task` -> `runDir/node<N>`
- `/ar-data` -> `/var/lib/ar/data`

规范要求：

- 共享状态写入 `/tasks`；
- 当前步骤临时产物写入 `/current-task`；
- 不得向镜像根文件系统写关键状态。

---

## 9. 状态机与任务生命周期规范

### 9.1 `taskId` 规则

- 由后端生成：`<UnixNano>_<随机后缀>`。
- `taskId` 在 `run/stop/resume/log` 全流程必须保持一致。

### 9.2 步骤状态

合法状态：

- `pending`
- `running`
- `success`
- `failed`
- `cancelled`

### 9.3 状态迁移

- 正常：`pending -> running -> success`
- 失败：`running -> failed`，并终止后续步骤
- 停止：`running/pending -> cancelled`

---

## 10. 构建规范（`pipeline build`）

### 10.1 基础命令

```bash
allrun pipeline build -p ./<pipeline>.template.json -t <pipeline-image>:<tag> -i ./images.txt
```

### 10.2 强制规则

- `-p` 必须指向 `*.template.json`。
- `images.txt` 每行一个镜像名，允许空行与 `#` 注释行。
- 构建必须基于 Dockerfile 第一条 `FROM`（或 `--from`）解析出的基础镜像。
- 基础镜像优先从本地 `--images-store-dir` 读取；不存在再远端拉取。

### 10.3 产物要求

流水线镜像内必须至少包含：

- `/<pipeline>.template.json`
- `/entrypoint.sh`
- `/images-raw/`（若需要携带子镜像）

`entrypoint.sh` 必须完成：

- 模板复制到容器挂载目录（`/pipelines`）；
- 子镜像导出到可被 loader 识别的位置（`/images` 或 `/images-store`）。

---

## 11. 加载规范（`pipeline load`）

### 11.1 支持来源

- 归档文件：`-i xxx.tar|xxx.tar.gz`
- 本地镜像：`--from-store <image>`

### 11.2 行为要求

- loader 必须验证镜像可执行入口（entrypoint 在 rootfs 存在）。
- 一次性容器执行后，模板必须落地到 `--pipelines-dir`。
- 子镜像必须成功导入 `--images-store-dir`；失败即整体失败。

---

## 12. 停止与恢复规范

### 12.1 停止

- 调用 `stop` 时：
  - 运行中步骤容器应被停止并清理；
  - `pending/running` 步骤标记为 `cancelled`；
  - 最终状态写回 `pipeline.json`。

### 12.2 恢复

- `resume` 从 `pipeline.json` 中第一个**非 success**步骤继续。
- 已 `success` 步骤不得重复执行。
- 恢复时仍按 DAG 拓扑顺序执行剩余步骤。

---

## 13. 步骤镜像与脚本开发规范

### 13.1 步骤镜像

- 每个步骤镜像必须可在非交互模式运行（`terminal=false`）。
- 必须包含其运行依赖（shell、工具链、脚本、离线制品）。
- 建议标准化目录：
  - `/scripts/remote/`
  - `/artifacts/bin|tarballs|images|manifests|checksums`

### 13.2 脚本编写

- 必须幂等，可重复执行不破坏已有状态（支持 resume/重试）。
- 必须显式返回非 0 退出码表示失败。
- 必须输出关键日志到 stdout/stderr（便于 `pipeline task log`）。
- 禁止在日志中明文打印密码、token、私钥内容。

---

## 14. 安全与合规规范

- 节点账号密码仅用于执行期，不得写入持久化明文文件。
- 如需跳过证书校验（`--tls-verify=false`），仅允许内网受控环境使用。
- 所有外部下载制品必须有版本锁定与校验和（`versions.lock` + checksum）。
- 生产流水线不得依赖未固定版本的远程在线资源。

---

## 15. 质量门禁（提测前必过）

### 15.1 模板校验

- JSON 合法；
- DAG 无环；
- 所有 `nodes` 引用合法；
- `step.name` 全局唯一；
- `image` 在镜像仓库或离线包中可解析。

### 15.2 执行校验

至少完成一次：

1. `load` 成功；
2. `run` 成功并全部 `success`；
3. `task log` 可读；
4. `stop` 能正确把未完成步骤置 `cancelled`；
5. `resume` 能从断点恢复。

### 15.3 可运维性校验

- `pipeline.json` 可读性良好（步骤名、状态、参数完整）；
- 日志文件命名符合 `<containerID>.stdout|stderr`；
- 失败时错误信息可定位到具体步骤。

---

## 16. 推荐开发流程

1. 在 `pipelines/<name>/` 初始化模板、`images.txt`、`Makefile`、`metadata`。
2. 先在小规模节点集验证模板渲染与 DAG 正确性。
3. 完成步骤镜像构建与离线制品封装。
4. 执行 `allrun pipeline build` 生成流水线镜像。
5. 执行 `allrun pipeline load` 验证模板与子镜像导入。
6. 执行 `allrun pipeline run -p <name> -n <nodes.json>`。
7. 验证 `stop/resume/log` 与失败回滚路径。
8. 更新 `design.md`、`versions.lock`、`manifest.json` 后发布。

## 17. 最小模板示例（规范版）

```json
[
  {
    "name": "print-plan",
    "image": "alpine:3.20",
    "entrypoint": "sh",
    "args": ["-c", "cat /tasks/pipeline.json"],
    "nodes": ["init-all"]
  },
  {
    "name": "init-all",
    "image": "my-step-init:1.0.0",
    "entrypoint": "bash",
    "args": ["-lc", "/scripts/remote/init.sh"],
    "env": [
      "NODES_MATRIX={{range $i, $n := .nodes}}{{if $i}};{{end}}{{$n.IP}}|{{$n.Port}}|{{$n.Username}}|{{$n.Password}}|{{$n.LabelsStr}}{{end}}"
    ]
  }
]
```

## 18. 变更控制要求

- 涉及字段模型、目录模型、状态机变更时，必须同步更新：
  - 本文档；
  - 对应 `design/*.md`；
  - `backend/pkg/pipeline` 与 GraphQL schema；
  - 至少一条真实流水线样例（建议 `pipelines/test`）。

- 任何破坏兼容性的变更必须在变更说明中明确：
  - 影响范围；
  - 迁移步骤；
  - 回滚方案。
